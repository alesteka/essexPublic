Use Cases

Levels of the use cases dictate development of a software, hence when considering all requirements we should be aware of certain levels. Common three levels are Sea level, Cloud and Fish level. The former depicts the interaction between system and its user/s whereas the task can be completed in a single sitting (e.g. borrowing a book or registering a new user). The Cloud level on the other side includes higher abstraction of the system and foremost it includes multiple sea levels (e.g lending a book requires identification of the customer and check of his unreturned books). Last level enhances the readability of the use cases. Tasks are divided into smaller parts for example finding a book or updating a record. However degree of caution has to be considered at this level, as we might get granular and consequently rigid system.

UML -> Common design technique used to develop systems/programs.

    Different types of models:

  • Behavioural: Captures the dynamic behaviour among objects used in the system.
  • Structural: This kind of models represent components of a system and relationships between them.
  • Architectural: Captures a high level perspective of a system.

    Common UML diagrams:

  • Class: The latter diagrams represent the relationship between objects that are used in a system. They help us to design and understand the properties and methods of a class.
  • Activity: Depicts the process and workflow of the business logic.
  • Sequence: Used for complicated projects, where we want to depict, how the messages are passed between objects.

Evolution of Programming Languages

  • Object-Oriented Programming: Main purpose of the following concept is the ability to create/simulate real world scenarios solutions, through object design paradigm.
  • Procedural Programming: Refers to a concept of procedural calls. Functionality of the program consists of calling the statements/functions.
  • Component-Based Programming: It consist of different software components, that are joined together, working as one entity.
  • Service-Oriented Software: It serves as a code that is mapped to a service. Infrastructure as a Code.

Core concepts

  • Abstraction: The process of simulating real world scenarios, whereas the idea is to make the objects understandable by a computer in a higher level. That way the designed solution can be easy to understand and consequently implemented as a software.
  • Encapsulation: Combining functions and data into a single unit. The data is not accessed directly, but through methods present inside the class. It uses private and public attributes, whereas getters and setters ae used to manipulate these attributes.
  • Inheritance: Sharing the methods between the objects. This way we are able to reuse already defined functionalities.
  • Composition: The functionality where attribute can be used as an object of another class.
  • Interfaces: The way in which objects communicate internally. Private/public designs dictate the communication between systems.

Java Inheritance, Composition

Dynamic (run time) Polymorphism: It happens when method gets overriden. The JVM knows which method to execute at runtime, based on the variable's class reference. (having two different objects with same class reference - JVM will execute the coresponding methods).
Static (compile time) Polymorphism: Having methods with same name, whereas Java compiler will know which method to reference based on the number or parameters passed as input. Also known as method overloading (early-binding)
